## 1. 코드 리뷰 요약 (현재 1차 코드)

**스택:** Express 4 + `pg`(PostgreSQL) + `express-session`, 빌드 없는 정적 HTML 3종(랜딩/어드민/로그인), Cloud Run. 단일 `server.js`(258줄).

**강점:** 디자인 토큰(CSS 변수)·반응형·카피 품질 우수, SQL 파라미터 바인딩 일관(인젝션 안전), Cloud Run 고려(trust proxy·SSL 분기·스키마 자동).

**문제점(심각도 순):**

| # | 심각도 | 이슈 | 위치 | 처리 시점 |
|---|---|---|---|---|
| 1 | 🔴 | 저장형 **XSS** — 공개 폼 입력을 어드민이 `innerHTML` 직접 렌더 → 어드민 세션 탈취 가능 | `admin.html:393,468` | P1 초입(이스케이프 즉시) |
| 2 | 🔴 | 어드민 인증 취약 — 단일 평문 비번, 해싱·rate limit·계정 없음 | `server.js:95-102` | **회원 시스템(P1)과 함께** |
| 3 | 🟠 | 기본 시크릿 폴백(`dev-secret`/`iroun2025!`) — env 누락 시 알려진 값으로 가동 | `server.js:26,97` | P1(부팅 시 필수 env 검증→하드 실패) |
| 4 | 🟠 | CSRF 방어 없음 + 쿠키 `sameSite` 미설정 | `server.js:25-33` | P1(인증 재구축 시) |
| 5 | 🟠 | 입력 검증·스팸 방어 부재(형식·화이트리스트·CAPTCHA·rate limit) | `server.js:49-61` | P1~P2 |
| 6 | 🟡 | CSV 인젝션(`=,+,@` 셀 수식 실행) | `server.js:206-232` | 백로그 |

## 2. 갭 분석 — 현재 코드는 PRD의 어디인가

현재 코드 ≈ PRD **STEP7(컨설팅 신청) + 백오피스(어드민)만** 구현. STEP1~6(AI 6단계)은 전무.

| PRD 구성 | 현재 | 관계 |
|---|---|---|
| STEP7 컨설팅 신청 | `submissions` + 상담폼 | ✅ 계승(원형) → `consulting_request`로 매핑 |
| 백오피스/CRM | 어드민 패널 | ✅ 확장 출발점 |
| 공개 랜딩(마케팅 자산) | `index.html` | 🔸 **카피 보존 + 디자인 리뉴얼** |
| STEP1 프로필 | name/phone/type/industry/interests | 🔸 얕은 버전 → 9세그먼트·다중태그·사업/기술/자금 확장 |
| 회원 인증·워크스페이스 | 없음(어드민 단일 비번뿐) | ❌ **최대 구조적 갭** |
| STEP2~6·매칭·크롤링연동·AI/RAG·결제·알림 | 없음 | ❌ 전부 신규 |

## 3. 확정 아키텍처 — 하이브리드

```
┌───────────────────────────────────────────────────────────────┐
│  공개 영역 (비로그인)                                          │
│   · 랜딩: 1차 카피(텍스트) 보존 + 디자인 스킬로 리뉴얼          │
│   · 마케팅 자산(성과수치·신뢰요소·CTA) 유지                    │
├───────────────────────────────────────────────────────────────┤
│  워크스페이스 (로그인)  ← 신규 구축                            │
│   · 회원 인증 + 마이페이지 + STEP1~6 + 결제                    │
│   · 좌우분할 에디터(STEP5)·실시간 섹션 스트리밍 등             │
├───────────────────────────────────────────────────────────────┤
│  백엔드 API (Express 확장 또는 분리)                          │
│   · 기존 /api/* 유지 + 신규 도메인 API(2장 컨텍스트별)        │
│   · 외부 크롤링 ACL(C5), AI 오케스트레이션(C12), 결제(C10)    │
└───────────────────────────────────────────────────────────────┘
```

**원칙**
- 공개 랜딩의 **카피는 자산** — 텍스트(헤드라인·페인포인트·성공사례·성과수치)는 살리고, **시각 디자인만 최신 디자인 스킬로 리뉴얼**.
- 복잡한 SaaS UI(워크스페이스)는 1차 정적 방식의 한계를 넘으므로 **적합한 프론트 구성**으로 신규 구축.
- 백엔드는 PRD 2장 **바운디드 컨텍스트** 경계를 따라 모듈화. 외부 크롤링은 **ACL**로 격리.
- 데이터는 PRD 4장 스키마가 청사진. `submissions` → `consulting_request` 매핑, `customer_profile` 중심 신규 도입, **마이그레이션 도구 도입**.

## 4. 단계별 실행 로드맵 (PRD Phase ↔ 코드 현실 정렬)

| Phase | 핵심 작업 | 코드 관점 진입점 |
|---|---|---|
| **P0 정지작업** | XSS 이스케이프 핫픽스, 필수 env 검증(하드 실패), 마이그레이션 도구·테스트 스캐폴딩 도입 | 기존 `admin.html`/`server.js` 최소 패치 |
| **P1 기반** | 랜딩 **디자인 리뉴얼**(카피 보존) + 회원 인증(해싱·세션·rate limit·CSRF) + 어드민 인증 재구축 + STEP1 프로필 빌더 + 마이페이지 골격 | 공개/워크스페이스 분리 시작 |
| **P2 매칭 연동** | 크롤링 ACL(C5) + `gov_program` 캐시 + STEP4 하드필터 + 매칭 결과 화면 | 핵심가치 MVP(AC-4.1/4.2/4.5) |
| **P3 AI 진단/생성 1차** | STEP2 아이템명 + STEP3 분석(YC/GStack 금지필터·TC-3.2) + STEP4 스코어링 고도화 | AI 오케스트레이션(C12) 도입 |
| **P4 사업계획서** | STEP5 RAG 파이프라인 + 결제(C10) | 매출 핵(AC-5.x, 게이팅) |
| **P5 슬라이드+컨설팅** | STEP6 + STEP7 고도화(전체 컨텍스트 첨부) + 패키지/구독 | 풀퍼널 완성 |
| **P6 고도화** | 매칭 가중치 피드백 루프 + 세그먼트 확장 + hwp | 운영데이터 정교화 |

> 우선순위 기준(PRD): **가장 빨리 매출로 연결되는 경로** = STEP4 매칭(무료 체험) → STEP5 사업계획서(유료 전환). 앞단 데이터 구조·추천 품질을 먼저 안정화.

## 5. 선결정 필요 (다음 논의)

- **R1(PRD):** 외부 크롤링 자격조건이 정형/비정형 여부 → 정형화 별도 공수 산정.
- **프론트 구성:** 워크스페이스를 어떤 방식으로 구축할지(현 Express 뷰 확장 vs 별도 SPA/SSR) — P1 착수 전 확정.
- **인증 방식:** 자체 이메일/비번 vs 소셜 로그인 vs 둘 다.
- **AI 모델·비용:** 사용 LLM과 API 약관(재학습 차단), RAG 인프라.

## 6. 즉시 다음 단계 후보

1. **랜딩 리뉴얼(P1 일부)** — 카피 보존 + 디자인 스킬로 시각 리뉴얼. 가시적 성과가 빠르고, 사용자가 선호 표명.
2. **P0 정지작업** — XSS 핫픽스 + env 검증 + 마이그레이션/테스트 스캐폴딩(기반 위생).
3. **데이터 모델 확정** — PRD 4장 스키마를 실제 마이그레이션으로 구체화(이후 모든 STEP의 토대).
